Earlier history: a Backup/Restore & SQLite Integrity — Production Hardening sprint (commit `d0b4826` on `main` — see §5.18) that audited the complete backup/export/import/restore system, SQLite foreign-key/cascade/index/transaction configuration, and crash-recovery behavior end-to-end, closing two real gaps found by direct inspection and adding one new defense-in-depth validation layer: (1) `backupRepository.restoreAllTables` — the single highest-blast-radius write in the app, since it wipes and repopulates all 11 tables — used `db.withTransactionAsync`, which expo-sqlite's own shipped source (`node_modules/expo-sqlite/build/SQLiteDatabase.js`, read directly against the installed SDK 54 version, not assumed from docs) explicitly documents as "not exclusive and can be interrupted by other async queries" issued elsewhere on the same shared connection — a concurrent export, or in principle any other screen's own data refresh, firing mid-restore could read a half-restored table or even get folded into the restore's own uncommitted transaction. Switched to `db.withExclusiveTransactionAsync` (opens a genuinely separate native connection with real mutual exclusion — any concurrent write, including a second overlapping restore, now blocks/fails against SQLite's own write lock instead of interleaving), with a `Platform.OS === 'web'` fallback to the prior behavior since `withExclusiveTransactionAsync` throws outright on web and this app's `npm run web` script, while present, is an unshipped Expo scaffold default (§2), not a target platform; (2) `insertRow` silently `return`ed (skipping the row entirely, no error) when a backup row had zero columns (e.g. `{}`), meaning a corrupted or hand-edited backup file could restore "successfully" while quietly dropping records — now throws, so the existing transactional catch-and-rollback path engages and the user sees an honest restore-failed message instead of silent data loss. New: `utils/backupValidation.ts` gained `hasReferentialIntegrity()`, checking every nullable/non-null foreign-key relationship in the schema (`birds.flockId`; `tasks.birdId`/`flockId`; `healthRecords.birdId`/`flockId`; `eggRecords.flockId`/`birdId`; `feedLogs.feedItemId`/`flockId`/`birdId`; `breedingPairs.maleBirdId`/`femaleBirdId`; `clutches.breedingPairId`/`flockId`; `candlingRecords.clutchId`; `hatchRecords.clutchId`) against the backup file's own id sets, entirely in memory, before any database write happens — restore itself still runs with `foreign_keys = OFF` (unchanged, needed for its cross-table delete/insert ordering), so without this check a backup with a dangling reference (e.g. a task pointing at a `birdId` that doesn't exist in the same file) would previously insert as a silent orphan rather than fail loudly. `isValidBackupData()`'s row-shape check was also tightened to require every row to be a non-null, non-array object with a numeric `id` (rejecting `{}`, arrays, and primitives up front — the same class of malformed input `insertRow`'s fix above defends against at a second layer). Both new checks were exercised against a 16-case scripted smoke test (valid backup, missing table key, empty-object row, string-typed row, dangling nullable and non-nullable foreign keys across 3 different tables, legitimately-null foreign keys, and schema-version comparisons — see §5.18) before being wired into `BackupRestoreScreen`'s `runRestore`, which now shows a distinct "Corrupted backup" alert (declining before any write) rather than either a generic failure or a silently-incomplete restore. Confirmed already correct, left unchanged: backup export (all 11 tables, `schemaVersion`, `exportedAt`); notification-ID nulling on restore; the delete/insert table ordering (already a valid parent-first topological sort per `db/migrations.ts`'s own FK declarations, so it doesn't strictly need `foreign_keys = OFF` for correctness — that pragma is kept only because a fresh `withExclusiveTransactionAsync` connection defaults to it anyway, and enabling FK enforcement mid-restore isn't reachable through expo-sqlite's public API without re-implementing its transaction helper from scratch); every other repository's transaction usage (`feedRepository`'s stock-adjusting `addFeedLog`/`updateFeedLog`/`deleteFeedLog` and `breedingRepository.createBirdsFromHatch` were already correctly wrapped in `withTransactionAsync`, and single-statement repository functions don't need explicit transactions since a lone `INSERT`/`UPDATE`/`DELETE` is already atomic under SQLite's own autocommit); and crash-recovery (WAL mode means an app kill mid-restore simply leaves an uncommitted transaction that's discarded on next launch — already true before this sprint, and on firmer footing now that restore runs on its own dedicated connection rather than the shared one). See §5.18. ## 1. Project Overview

**Flockora** is a mobile-first flock care, hatching, breeding, and bird management app for backyard poultry keepers and hobby breeders. Tagline: *"Care. Hatch. Breed. Protect."*

- **Core purpose:** Replace notebooks/spreadsheets for people who keep and breed backyard birds (chickens, quail, ducks, turkeys, geese, guinea fowl, pheasants, peafowl) with a warm, simple, camera-and-voice-first mobile app.
- **Target users:** Western (US/UK/Canada/Australia) backyard poultry keepers with mixed flocks, hobby breeders, small breeders outgrowing spreadsheets, households sharing bird care, and people who hatch birds at home. Explicitly **not** targeted at large industrial poultry operations.
- **Main product concept:** Four pillars — **Today** (what needs attention now), **Care** (daily flock stewardship: birds, eggs, feed, health), **Hatch** (incubation, candling, hatch tracking), **Breed** (pairings, lineage, fertility/hatch performance) — plus a **Protect** pillar for health awareness. Full philosophy, rules, and design language are defined in [`PRODUCT_CONSTITUTION.md`](PRODUCT_CONSTITUTION.md) at the repo root — **read that file first**, it is the authoritative product spec this codebase implements phase by phase.
- **Guiding UX principle (constitution §2):** "Less typing. More understanding." AI prepares, the human confirms, the system calculates — data is never silently upgraded from AI-proposed to verified fact.

The repository root now contains two independent projects: `flockora-app/` (the Expo mobile client, detailed below) and, as of Phase 3 Sprint 3.1, `flockora-backend/` (a minimal Node/Express service — see §14). They are not a monorepo with shared tooling, just two sibling directories in one git repository; the client has no dependency on the backend today (it doesn't call it yet).

## 4. Database and Data Structure

Single local SQLite database, file name `flockora.db`, managed entirely through [`src/db/migrations.ts`](flockora-app/src/db/migrations.ts). **No cloud sync, no server-side database exists.**

- **Migration pattern:** one file, an exported `DATABASE_VERSION` constant (currently **10**; exported as of the Backup / Export / Import sprint so `backupValidation.ts` can compare a backup file's `schemaVersion` against it — previously module-private, since nothing outside `migrations.ts` needed it), and sequential `if (currentVersion === N) { ...; currentVersion = N+1; }` blocks run against `PRAGMA user_version`. Each block is idempotent (`CREATE TABLE IF NOT EXISTS` / additive `ALTER TABLE ... ADD COLUMN`, or a create-copy-drop-rename table rebuild when a constraint itself must change — see v8→v9 below). `PRAGMA foreign_keys = ON` is set unconditionally on every app start, and `PRAGMA journal_mode = WAL` was set during the v0→v1 migration.
- **To add a new table/column:** bump `DATABASE_VERSION`, append a new `if (currentVersion === 10) { ...; currentVersion = 11; }` block — never edit prior blocks (they may already have run against real user devices, even though this app has no production users yet).
- **v8→v9 (Care & Health Management sprint):** SQLite can't relax a column's `NOT NULL` constraint via `ALTER TABLE`, so this migration rebuilds `health_records` (create `health_records_new` with the new shape → `INSERT ... SELECT` every existing row, with `flockId`/`time` defaulting to `NULL` → `DROP TABLE health_records` → `RENAME TO health_records` → recreate indexes) instead of a plain `ADD COLUMN`. This is the first migration in this codebase to use the rebuild pattern rather than additive `ALTER TABLE`; any future migration that needs to change a constraint (not just add a column) should follow the same shape.
- **v9→v10 (Vet Document Attachment sprint):** a plain additive `ALTER TABLE health_records ADD COLUMN documentUri TEXT;` — no constraint changed, so the v8→v9 rebuild pattern wasn't needed here.

### 5.10 Backup / Export / Import (`BackupRestoreScreen`, `MoreStack`)
- Files: `screens/BackupRestoreScreen.tsx` (new), `db/repositories/backupRepository.ts` (new), `services/backupFileService.ts` (new), `types/backup.ts` (new), `utils/backupValidation.ts` (new), `navigation/moreTypes.ts` (`BackupRestore` route added), `navigation/MoreStack.tsx` (route registered), `screens/ReportsScreen.tsx` (new header `IconButton` linking to it), `db/migrations.ts` (`DATABASE_VERSION` now exported, previously module-private), `db/repositories/index.ts` (barrel export added).
- **Export:** `backupRepository.exportAllTables(db)` runs `SELECT * FROM <table>` (via `Promise.all`) against all 11 tables — `flocks`, `birds`, `tasks`, `health_records`, `egg_records`, `feed_items`, `feed_logs`, `breeding_pairs`, `clutches`, `candling_records`, `hatch_records` — and wraps the results in a `BackupData` object (`{ schemaVersion, exportedAt, ...one array per table }`). `backupFileService.writeAndShareBackup()` then `JSON.stringify`s it, writes it to `Paths.cache` via `expo-file-system`'s `File` class, and hands it to `expo-sharing` — the exact same three-step pattern `eggExport.ts` already established for CSV, just with a JSON payload and `.json` extension/MIME type instead.
- **Import:** `backupFileService.pickBackupFile()` wraps the new `expo-document-picker` dependency (`DocumentPicker.getDocumentAsync`), returning a discriminated `PickBackupFileOutcome` (`success | canceled | error`) — the same outcome-object shape `imagePickerService.ts`'s `PickPhotoOutcome` already established for camera/gallery picks. Deliberately filters on `type: '*/*'` rather than a JSON MIME type, since Android file providers inconsistently report `.json` files as `text/plain`/`application/octet-stream`; the picked file's actual content is validated after reading instead of trusted by MIME type. `readBackupFile()` reads the picked URI's text via `new File(uri).text()` and `JSON.parse`s it.
- **Validation before restore:** `utils/backupValidation.ts`'s `isValidBackupData()` (a pure type-guard — checks `schemaVersion`/`exportedAt` types, that all 11 table keys are arrays, and, as of §5.18, that every row in every array is a non-null, non-array object with a numeric `id`) rejects anything that isn't a real Flockora backup with a clear "Not a valid backup" alert. `checkSchemaCompatibility()` compares the backup's `schemaVersion` against the running app's `DATABASE_VERSION` (now exported from `migrations.ts` for exactly this purpose) and refuses to restore a backup newer than the current schema ("Update required") rather than risk inserting into columns that don't exist yet — a backup *older* than the current schema is accepted, since every additive column introduced by a migration so far is nullable, so `INSERT`ing a row missing that column just leaves it `NULL` (SQLite's normal default-on-omission behavior). As of §5.18, `hasReferentialIntegrity()` additionally checks every foreign-key column in the backup against the id sets of its own referenced tables and rejects a backup with a dangling reference ("Corrupted backup") before any table is touched — restore itself runs with `foreign_keys = OFF`, so without this check a stale/tampered reference would otherwise insert as a silent orphan rather than fail.
- **Restore is destructive and transactional:** `backupRepository.restoreAllTables(db, data)` deletes every table in child-first order, then re-inserts every row from the backup in parent-first order using a generic column-driven `INSERT` (built from each row's own JSON keys, so it tolerates schema drift automatically). IDs are inserted explicitly (SQLite's normal, supported behavior for an `INTEGER PRIMARY KEY AUTOINCREMENT` column) so every foreign-key relationship in the restored data stays internally consistent. **As of the Backup/Restore & SQLite Integrity — Production Hardening sprint (§5.18), the whole delete+insert runs inside `db.withExclusiveTransactionAsync`** (a genuinely separate, mutually-exclusive connection — not the non-exclusive `db.withTransactionAsync` used originally, which expo-sqlite's own source documents as interruptible by other async queries on the same shared connection), with a web-platform fallback to the original `withTransactionAsync` path since the exclusive variant isn't supported there. `foreign_keys` is left at that fresh connection's own default (`OFF`) rather than explicitly toggled, since `TABLES`' parent-first order is already a valid topological sort either way. A failure partway through — including a now-rejected malformed row (§5.18) — rolls back to the pre-restore state rather than leaving the database half-wiped.
- **Notification IDs are deliberately cleared on restore, not carried over:** `tasks.notificationId`, `health_records.notificationId`, `feed_items.notificationId`, and `clutches.candlingNotificationId`/`hatchExpectedNotificationId`/`hatchDueNotificationId` are forced to `NULL` for every restored row, because these reference `expo-notifications`-scheduled OS notifications that are not themselves part of this SQLite-only backup — restoring a stale ID would just point at a notification that no longer exists (or never existed on this device). This is a known, disclosed limitation, not a bug: the restore-complete alert tells the user reminders weren't restored and that reopening/re-saving a task or care record will reschedule its notification. ### 5.18 Backup/Restore & SQLite Integrity — Production Hardening sprint

A full audit-and-harden pass over the backup/export/import/restore system (§5.10), SQLite schema (foreign keys, cascade behavior, indexes — `db/migrations.ts`), and crash-recovery behavior, not a rewrite. See the header summary above for the complete list of what was found and fixed; this section is the file-level reference.

- **Files changed:** `src/db/repositories/backupRepository.ts` (`restoreAllTables`, `insertRow`), `src/utils/backupValidation.ts` (`isValidBackupData` tightened, new `hasReferentialIntegrity`), `src/screens/BackupRestoreScreen.tsx` (wires the new check into `runRestore`).
- **Restore transaction isolation:** `restoreAllTables` now uses `db.withExclusiveTransactionAsync` (a genuinely separate native connection, per `node_modules/expo-sqlite/build/SQLiteDatabase.js`) instead of `db.withTransactionAsync`, which expo-sqlite's own installed source documents as non-exclusive and interruptible by other async queries on the same connection. Since restore is the one operation that touches every table, this was the app's single biggest atomicity gap: a concurrent export or any other screen's data refresh firing mid-restore could previously read a half-restored table, and a second overlapping restore could interleave with the first instead of being rejected. The prior explicit `PRAGMA foreign_keys = OFF` / `ON` toggle around the transaction was removed — a freshly opened connection (which `withExclusiveTransactionAsync` always creates) already defaults to `foreign_keys = OFF`, matching the old behavior, and `TABLES`' parent-first order (verified against every `ON DELETE` clause in `db/migrations.ts`) is already a valid topological sort for both the reverse-order deletes and as-listed inserts, so no other change to the delete/insert logic was needed. `Platform.OS === 'web'` falls back to the previous `withTransactionAsync` path, since `withExclusiveTransactionAsync` throws unconditionally on web and `npm run web` — while a live script — is an unshipped Expo scaffold default (§2), never a target platform for this app; breaking it silently would have been its own regression.
- **Malformed-row handling:** `insertRow` previously `return`ed silently when a backup row had zero columns (e.g. `{}`), meaning such a row was dropped with no error and no rollback while the restore still reported success. It now throws, so the existing transactional catch rolls back the whole restore and the screen shows its existing "Restore failed... No changes were made" alert — an honest failure instead of silent, partial data loss.
- **New pre-write validation (`backupValidation.ts`):** `isValidBackupData` now requires every row in every table array to be a non-null, non-array object with a numeric `id` (previously only checked that each table key held an array, at all — a `["oops"]` or `[{}]` payload passed before). New `hasReferentialIntegrity(data)` walks every foreign-key relationship declared in `db/migrations.ts` (16 columns across 9 of the 11 tables) and confirms each non-null value resolves to a real `id` elsewhere in the same backup file, entirely in memory, before `restoreAllTables` is ever called — this is what actually closes the "no orphan records" requirement, since restore's own `foreign_keys = OFF` means SQLite itself won't catch a dangling reference at insert time. `BackupRestoreScreen.runRestore` calls it right after the existing schema-compatibility check and shows a new "Corrupted backup" alert if it fails, before any table is touched.
- **Verified, not changed:** export already captures all 11 tables plus `schemaVersion`/`exportedAt` (`backupRepository.exportAllTables`, unchanged); notification-ID nulling on restore (`NOTIFICATION_ID_COLUMNS`, unchanged, still correct); every index and `ON DELETE` clause in `db/migrations.ts` matches the FK-behavior table in §4 exactly; every other repository's transaction usage — `feedRepository.addFeedLog`/`updateFeedLog`/`deleteFeedLog` (stock-quantity + log-row writes) and `breedingRepository.createBirdsFromHatch` (multi-bird creation guarded by the `hatch_records.birdsCreated` flag) were already correctly wrapped in `db.withTransactionAsync`; every other repository function is a single `INSERT`/`UPDATE`/`DELETE` statement, which is already atomic under SQLite's own autocommit and needs no explicit transaction. Read-then-write validation patterns that exist elsewhere (e.g. `breedingRepository.saveHatchRecord`'s create-or-update-by-`clutchId`, `addCandlingRecord`'s clutch-total check) are pre-existing, single-JS-thread-appropriate patterns for a local-only, single-user app and were left untouched, consistent with this sprint's "don't rewrite working code" instruction.
- **Crash-recovery:** unchanged in mechanism (WAL journal mode, set once at v0→v1 migration, means an app kill mid-restore simply leaves an uncommitted transaction that SQLite discards on next open — this was already true), but now on firmer footing since restore runs on its own dedicated connection rather than sharing the app's one main connection with every other in-flight query.
- **Verification performed this sprint:** the new validation logic (`isValidBackupData`, `hasReferentialIntegrity`, `checkSchemaCompatibility`) was compiled standalone and exercised against a 16-case scripted smoke test covering a well-formed backup, `null`/array/string top-level values, a missing table key, an empty-object row, a string-typed row, dangling foreign keys on 3 different tables (a nullable `tasks.birdId`, a non-nullable `breedingPairs.maleBirdId`/`femaleBirdId` pair, and `candlingRecords.clutchId`), legitimately-`null` nullable foreign keys (confirmed not to false-positive), and all three schema-version-compatibility branches — all 16 passed. `npx tsc --noEmit` (clean), `npx tsc --noEmit --noUnusedLocals --noUnusedParameters` (clean, zero findings), `expo-doctor` (18/18), `expo export --platform android` (succeeded, run twice — once after the initial fix, once after adding the web-platform fallback), `npm audit` (identical 12 moderate findings to the documented baseline — zero new, no dependency touched), and a security-regression grep (`console.*`, `EXPO_PUBLIC_*`/`process.env`, `fetch`/`axios`/`XMLHttpRequest`, `http(s)://`, secret/token/password patterns) across the full diff — zero matches. No automated UI/on-device test of the restore flow itself was performed (per §10, no test suite/emulator exercise was available in this session) — stated explicitly rather than claimed as verified.

**1. Virtualized two more unbounded lists (`PulseHomeScreen`, `ClutchHistoryScreen`).**
Both screens rendered their full result set via `ScrollView`+`.map()` — `PulseHomeScreen`'s cross-flock health-record search and `ClutchHistoryScreen`'s all-time clutch history, both of which grow without bound over the life of the app (years of care records / breeding seasons), exactly the same shape `EggHistoryScreen`/`FeedHistoryScreen` were converted to `FlatList` for in an earlier sprint (§5.15) — these two were simply missed at the time. **Before:** every row (every matching health record / every clutch ever logged) mounted and rendered on every screen visit, regardless of how many exist. **After:** `FlatList`'s default windowing renders only the on-screen rows plus a small overscan buffer, independent of total row count. A `FlatList` conversion would need that header content moved into `ListHeaderComponent` (still scrolls with the list, preserving that behavior) — safe — but the list section's own `FadeInUp` entrance animation, which today only mounts (and only fades in) once the bird data is ready and non-empty, has no clean equivalent once the header and list share one `FlatList` shell: wrapping the whole `FlatList` in `FadeInUp` would fade in the header too (starting immediately on screen mount, before data arrives) rather than just the list once ready — a small but real visible-behavior change this sprint's "preserve all existing behavior" rule doesn't allow trading for a virtualization win. Left as-is; a future sprint that's explicitly scoped to include minor animation-timing changes could revisit this.

**2. Stabilized `FlatList` `renderItem` (all 4 history screens now).**
`EggHistoryScreen`/`FeedHistoryScreen`'s existing `FlatList`s defined `renderItem` as an inline arrow directly in JSX — a new function identity every render. `FlatList`'s internal `VirtualizedList`/`CellRenderer` layer uses `renderItem`'s referential stability (along with each item's own identity) to decide whether a given on-screen cell needs to be re-invoked at all; an unstable `renderItem` forces it to treat every render as "might have changed," undermining the exact optimization virtualization is supposed to provide. Extracted `renderItem`, `keyExtractor`, and the delete handler each closes over to `useCallback` in all four screens (the two pre-existing `FlatList`s plus the two newly converted in item 1), so an unrelated re-render of the screen (e.g. a sibling filter chip's local state changing) no longer forces `FlatList` to reconsider cells whose underlying row data hasn't changed.

**3. Removed a duplicate database read on every mount and filter change (4 hooks).**
`useEggHistory`, `useHealthRecords`, `useFeedHistory`, and `useClutches` each had:
```ts
useEffect(() => { refresh(); }, [refresh]);
useFocusEffect(useCallback(() => { refresh(); }, [refresh]));
```
Both effects depend on the same `refresh` callback. **Before fixing this, I read `@react-navigation/core`'s actual shipped `useFocusEffect` source** (`node_modules/@react-navigation/core/lib/module/useFocusEffect.js`) rather than assuming — its internal `React.useEffect` has `[effect, navigation]` as its dependency array, and its body runs the callback immediately if `navigation.isFocused()` is already true (true at initial mount for the active screen), plus re-runs on every subsequent `'focus'` event and every time `effect`'s identity changes. That means `useFocusEffect` alone already fires on: initial mount, every subsequent focus, and every filter change (since `refresh`'s own `useCallback` deps are the primitive filter fields, so its identity changes exactly when a filter changes) — the plain `useEffect` sitting next to it fires on the exact same triggers with no additional coverage, so it was a pure duplicate. **Before:** 2 full `SELECT`-and-filter reads back-to-back on every screen mount and every filter/search keystroke settling. **After:** 1. Removed the plain `useEffect` from all four hooks, leaving only `useFocusEffect` — which is also now the *only* refresh pattern used anywhere in the hooks layer (`useBirds`, `useFlocks`, `useTodayTasks`, `useCareDashboard`, and every other hook already only used `useFocusEffect`; these four were the sole outliers). Updated the stale rationale in §9's "Filtered-list hooks pattern" bullet, which had attributed the second effect to "avoiding stale-closure/reference-equality refresh bugs" — that protection actually comes from `refresh`'s own primitive-field dependency array (unchanged by this fix), not from having two separate effects call it.

### 5.20 UI Correctness — Double-Submission Prevention on Save/Confirm Buttons — Production Hardening sprint

A resumed sprint (the prior session was interrupted mid-way by the hourly session limit; the working tree already contained a complete, self-consistent fix when this session started, so it was verified and finished rather than redone) that closed one real correctness bug: every form screen's Save/Confirm button visually dimmed itself while its async save was in flight (`style={saving ? styles.disabled : undefined}`), but the underlying `Pressable` was never actually disabled, so a fast double-tap — plausible on any real device, especially a slower one or one under load — could fire `handleSave`/`handleCreate`/`handleConfirm` a second time before the first call's `setSaving(true)` had a chance to re-render the button. For every affected screen that's a database write, so a double-tap could silently create a duplicate row (a second bird, task, health record, egg log, feed item, breeding pair, clutch, or candling/hatch record) from one user action, with no error and no visible sign anything went wrong.

- **Files changed:** `src/components/PrimaryButton.tsx`, and all 13 screens using the `saving ? styles.disabled : undefined` pattern: `AddEditBirdScreen`, `AddEditBreedingPairScreen`, `AddEditCandlingRecordScreen`, `AddEditClutchScreen`, `AddEditEggRecordScreen`, `AddEditFeedItemScreen`, `AddEditFlockScreen`, `AddEditHatchRecordScreen`, `AddEditHealthRecordScreen`, `AddEditTaskScreen`, `CreateBirdsFromHatchScreen`, `LogFeedUsageScreen`, `ReviewConfirmBirdDetailsScreen`.
- **`PrimaryButton`:** gained a new optional `disabled?: boolean` prop, wired straight to the underlying `Pressable`'s own `disabled` (the actual mechanism that blocks `onPress` from firing — RN's `Pressable` does not infer this from `style`), plus `accessibilityState={{ disabled: !!disabled }}` so screen readers announce the state change rather than only a sighted user seeing the dimmed style. The existing `style={saving ? styles.disabled : undefined}` visual treatment at every call site was left as-is (it's still correct, just no longer sufficient on its own) — this fix adds the missing enforcement, it doesn't replace the existing visual cue.
- **Call sites:** each of the 13 screens' Save/Confirm `PrimaryButton` gained `disabled={saving}` alongside its pre-existing `style={saving ? styles.disabled : undefined}`. Verified each screen's `setSaving(true)` call happens synchronously at the start of its `handleSave`/`handleCreate`/`handleConfirm`, before any `await` — so the very next render after the first tap already has the button disabled, closing the race rather than narrowing it.
- **Scope verification:** `grep -rn "saving ? styles.disabled : undefined" src/screens` returned exactly these 13 files, matching the working tree's diff file-for-file — confirming the interrupted session's in-progress work was already complete and internally consistent (no call site missed, no call site touched that wasn't already using this exact anti-pattern). Other `PrimaryButton` call sites with their own async actions (e.g. `BackupRestoreScreen`'s "Choose Backup File") were checked and don't use this pattern — most either navigate immediately (no in-flight async gap to double-fire into) or already guard re-entry a different way — and were correctly left untouched as out of this sprint's scope.
- **No backend, no analytics, no cloud sync, no new dependency, no new platform permission, no `app.json` change, no database schema change** — a pure client-side UI-correctness fix confined to one shared component and its existing call sites.
- **Validation performed this sprint:** `npm run typecheck` (`tsc --noEmit`, clean), `expo-doctor` (18/18, unchanged baseline), `expo export --platform android` (succeeded), `npm audit` (identical 12 moderate findings to the documented baseline — zero new, no dependency touched), and a security-regression grep (`console.*`, `EXPO_PUBLIC_*`/`process.env`, `fetch`/`axios`/`XMLHttpRequest`, `http(s)://`, secret/token/password patterns) across the full diff — zero matches beyond the already-documented, already-compliant `EXPO_PUBLIC_BACKEND_URL` reference and backend-fetch call site. No automated test suite exists in this repo (per §10), so the fix was verified by static review of every call site's `setSaving`/`await` ordering, not by interactively double-tapping on a device in this session — stated explicitly rather than claimed as verified.

### 5.21 Async State, Lifecycle & Concurrency — Production Hardening sprint

A read-then-fix audit across 5 areas named by this sprint's own brief — async save/delete handlers, React lifecycle safety, SQLite concurrency, rapid-user-action sequences, and loading/error-state correctness — not a rewrite. Every hook in `src/hooks/` (32 files) and every save/delete handler in every screen/modal was read in full. See the header summary above for the short version; this section has the full file-level detail.

- **Files changed:** `src/components/FlockManagerModal.tsx`, `src/hooks/useBirds.ts`, `src/hooks/useHealthRecords.ts`, `src/hooks/useEggHistory.ts`, `src/hooks/useFeedHistory.ts`, `src/hooks/useClutches.ts`.

**1. Duplicate flock creation on double-tap (`FlockManagerModal.tsx`).**
`handleCreate` had no re-entrancy guard at all — no `creating` state, no disabled prop on its inline "+" `Pressable`, and the same handler is also wired to the flock-name `TextInput`'s `onSubmitEditing`. A fast double-tap on "+", or tapping "+" while also pressing Enter/Done on the keyboard, invoked `flockRepository.create(db, { ...createEmptyFlockInput(), name })` a second time before the first call's `await` resolved and `setNewFlockName('')` cleared the input — both calls read the identical trimmed name, so the result was two duplicate flock rows from one user action, no error shown. This is the same class of bug §5.20 fixed for every screen-level Save/Confirm button, but that sprint's own grep (`saving ? styles.disabled : undefined`) was scoped to a pattern this modal never used — it has its own inline create flow, not a full-screen form. Verified as the last standing gap by grepping every `repository.create()`/`.add*()` call site across `src/screens` and `src/components`: all 13 screen-level sites were already covered by §5.20; this was the only one left. **Fix:** added `const [creating, setCreating] = useState(false)`, an `if (creating) return;` guard at the top of `handleCreate`, a `try/finally` around the `await flockRepository.create(...)` call, and `disabled={creating}` (plus `accessibilityState={{ disabled: creating }}`) on the "+" `Pressable` — mirroring the exact `creating`/`exporting`/`restoring`-boolean-guard pattern `BackupRestoreScreen` already used for its own async actions.

**2. Stale/out-of-order refresh responses in 5 argument-driven hooks.**
`useBirds`, `useHealthRecords`, `useEggHistory`, `useFeedHistory`, and `useClutches` each have a `refresh()` whose identity (and thus whose `useFocusEffect`-driven re-fire) depends on an argument that can change while the same screen instance stays mounted: `useBirds(flockId)`'s `flockId` changes on every flock-chip tap in `FlockHomeScreen`; the other four's `filters` object is rebuilt from `searchText`/`date`/etc. state that changes on every keystroke in an undebounced search box on `EggHistoryScreen`/`FeedHistoryScreen`/`PulseHomeScreen` (via `useHealthRecords`)/`ClutchHistoryScreen`. None of the five had any guard against two overlapping `refresh()` calls resolving out of order. **This is not the duplicate-`useEffect` bug §5.19 already fixed** — that was the *same* trigger firing the *same* refresh twice; this is *different* triggers (old filter vs. new filter) racing each other, and re-reading the current hook source confirmed the §5.19 fix is still correctly in place (only one `useFocusEffect`, no stray `useEffect`) in all five. §5.18's own investigation of expo-sqlite's installed source (`node_modules/expo-sqlite/build/SQLiteDatabase.js`) already established, by direct inspection rather than assumption, that queries against the shared connection are not guaranteed to serialize/complete in submission order — that's precisely why `backupRepository.restoreAllTables` needed `withExclusiveTransactionAsync` instead of the non-exclusive `withTransactionAsync`. The same non-ordering-guarantee applies here: if a user types quickly enough that a second keystroke's DB read is dispatched before the first keystroke's read has resolved, and the first (now-stale) read happens to resolve second, its unconditional `setState` call at the end of `refresh()` would silently overwrite the correct, current-filter results with results for a filter the user has already moved past — no error, no visual indication, just wrong data on screen until the next successful refresh. **Fix:** each of the 5 hooks gained a `const requestIdRef = useRef(0)`; `refresh()` now increments it and captures the value locally at the start (`const requestId = ++requestIdRef.current`), and skips both `setState` calls at the end if `requestId !== requestIdRef.current` (i.e. a newer `refresh()` call has started since). This is the standard, minimal "ignore stale response" pattern — no behavior change for the winning (latest) call, and the loading/empty/error UI shape is untouched. **Deliberately scoped to only these 5 hooks**, not all 32: the other 27 either take no argument at all (`useFlocks`, `useTodayTasks`, `useCareDashboard`, every `*Dashboard`/`*Statistics` hook — re-triggered only by navigation focus events, not by rapid in-screen interaction, since there's no argument for the user to change while mounted) or take a fixed route-level `id` (`useBird`, `useTask`, `useHealthRecord`, `useEggRecord`, `useFeedItem`, `useFeedLog`, `useClutch`, `useBreedingPair`, `useCandlingRecords`, `useHatchRecord`, `useHatchRecordById`, `useHealthStats`) that in this app's navigation pattern never changes without a fresh screen instance being pushed. Applying the same guard there would be unproven, blanket work with no concrete bug behind it — exactly the "broad refactor" this sprint's own rules exclude.

**Reviewed in depth, deliberately left unchanged (each with a specific reason, not by omission):**
- **Missing try/finally, or navigation before DB work completes** — every one of the 13 Add/Edit screens' `handleSave` (§5.20's own file list) and every delete handler (`BirdProfileScreen`, `ClutchDetailScreen` ×2, `HealthRecordDetailScreen`, `FeedInventoryScreen`, `TaskDetailScreen`, `BackupRestoreScreen`) was read in full: every one wraps its DB work in try/finally with the `saving`/`exporting`/`restoring` reset in `finally`, and every `navigation.goBack()`/`.replace()` call sits after the awaited repository call, never before or racing it.
- **Destructive (delete) actions firing more than once** — every delete flow in the app routes through `confirmDestructive()` → a native `Alert.alert`, which the OS itself only allows one tap/dismissal on; unlike an in-screen `Pressable`, it isn't vulnerable to the fast-double-tap class of bug this sprint targeted.

Already the most defensively written async flow in the codebase; used as the model for this sprint's `FlockManagerModal` fix.
- **State updates after unmount** — this app is on React 19.1 (§2); React has not warned on, or misbehaved from, a `setState` call after unmount since React 18's concurrent-rendering model made it a silent no-op. Confirmed no class-component `setState` exists anywhere in `src/` (`grep`, zero matches). Not a real bug category for this codebase's React version, and no fix was applied purely to add unmount-guard ceremony with no functional effect.
- **Rapid task completion/reopen (`toggleTaskCompletion`, `TaskRow`'s checkbox)** — has no explicit re-entrancy guard, but a double-tap has no observable harmful effect: `taskRepository.completeTask`/`.reopenTask` are idempotent `UPDATE`s (setting the same `completed`/`completedAt` value twice is a no-op), and §5.17 already gave every `scheduleX` notification call a deterministic per-entity `identifier` (e.g. `task-<id>`), so a double-tap's second `scheduleTaskNotification` call structurally replaces the same OS notification rather than creating a duplicate. ## 8. Important Technical Decisions

- **Local-only SQLite, no backend, built phase-by-phase:** Directly follows constitution §19 ("The application should not be built all at once... developed phase by phase") and §15's explicit instruction not to integrate external services until told to. Each git commit corresponds to one "Sprint" of `PRODUCT_CONSTITUTION.md`-driven work (see `git log`).
- **Repository pattern with `db` as first parameter:** Every repository function takes `SQLiteDatabase` explicitly rather than importing a singleton — keeps data access testable and avoids hidden global state. Reason inferred from consistent pattern across all 7 repository files; not separately documented in-repo.
- **FK delete-behavior split (CASCADE vs SET NULL):** Consistently reasoned in migration comments-equivalent structure (see §4 table) — CASCADE for genuinely dependent child records, SET NULL for records that should survive their parent's deletion to preserve historical data, directly implementing constitution §17 ("Breeding and hatch lineage must preserve historical relationships").
- **Derived state never stored:** Incubation phase, task overdue/completed-today status, and similar UI states are computed on read from stored dates rather than persisted as columns — avoids state going stale. Pattern is consistent across `taskSchedule.ts` and `breedingCalc.ts`.
- **One shared `notificationService.ts` file, extended per module rather than duplicated:** All local notification scheduling logic lives in a single file, growing by one `scheduleXReminder`/`syncXReminder` pair per feature. Reason: keeps `expo-notifications` trigger-construction logic (a real source of subtle bugs across trigger types) in one auditable place.
- **`SQLiteProvider` with `useSuspense` + `Suspense` fallback in `App.tsx`:** Chosen (per expo-sqlite's current API) over manually managing a loading boolean; matches the new class-based expo-sqlite API for SDK 54.
- **Sex classification via bounded term list, not free-text NLP (`birdSex.ts`):** `classifyBirdSex()` matches against a small, code-verified list of terms actually produced elsewhere in the app (`Hen`, `Female`, `Gander`, `Male`, `Unknown`, etc.) rather than attempting general natural-language parsing, and degrades to a non-blocking "unknown" rather than erroring — reduces false-positive validation blocks on ambiguous data entry.
- **`expo-file-system`'s new `File`/`Directory`/`Paths` class API used for CSV export**, not the legacy path-string API — matches SDK 54's current API surface.

## 9. Important Product and UI Behaviour

- **Feature-hub navigation pattern:** Every major feature (Egg, Feed, Breeding) follows the same shape — an entry card on `FlockHomeScreen` → a dedicated hub screen with its own stats/quick-actions → Add/Edit forms and History/Detail screens, all registered as additional routes inside the existing `FlockStack` (never a new bottom tab, per constitution §13's 5-tab limit). `TodayScreen` gets a compact 4-card "preview" summary section per module — visually and functionally distinct from (shallower than) the module's own full hub screen.
- **Confidence states (constitution §4/§8):** `HIGH`/`LIKELY`/`UNSURE` are a first-class type (`ConfidenceLevel` in `src/types/onboarding.ts`) rendered via the `ConfidenceBadge` component, currently only exercised by the mocked onboarding AI-analysis flow.
- **Today dashboard (`TodayScreen`):** Single scrollable screen combining: today's task progress bar + task list, then stacked summary-card sections for Flock Health, Egg Production, Feed, and Breeding — each section pulls from its own `useXDashboard`/`useXDashboardStats` hook, all independent, all refreshing via `useFocusEffect`.
- **Filtered-list hooks pattern:** Hooks that take filter objects (e.g. `useHealthRecords(filters)`, `useClutches(filters)`) key their `refresh` callback on primitive filter fields (`filters.searchText`, `filters.flockId`, etc.), not the `filters` object reference itself — the object is rebuilt fresh every render at the call site, so keying on the reference would refetch on every render regardless of whether any field actually changed. As of the Performance & Scalability — Production Hardening sprint (§5.19), refetching is driven solely by `useFocusEffect(useCallback(() => refresh(), [refresh]))` — verified against `@react-navigation/core`'s actual source that this alone fires on mount, on every focus, and on every `refresh` identity change (i.e. every filter change), matching every other hook in the codebase (`useBirds`, `useFlocks`, `useTodayTasks`, etc.). A previously-present second `useEffect(() => refresh(), [refresh])` was removed as a confirmed duplicate — it fired on the exact same triggers with no unique behavior of its own.
- **Error handling convention:** Business-rule violations are custom typed `Error` subclasses thrown from repositories (e.g. `InsufficientStockError`, `SameBirdPairingError`, `InvalidSexPairingError`, `CandlingCountExceedsClutchError`, `ClutchTotalReductionError`, `DuplicateBirdCreationError`), caught specifically in the calling screen and shown as a friendly `Alert`, rather than generic try/catch-and-log.
- **Notification reconciliation:** Every editable entity with a scheduled reminder uses a cancel-then-reschedule pattern (`syncXReminder(entity, previousNotificationId)`) on every save, so stale notifications never accumulate when due dates/settings change. As of the Notification & Reminder System — Production Hardening sprint (§5.17), every scheduled notification also carries a deterministic `identifier` (e.g. `task-<id>`) as a structural second line of defense against duplicates, plus a `categoryIdentifier`/`data.type` pair used for iOS categories and cross-platform deep-linking respectively.
- **Notification deep-linking:** Tapping any task/health/feed/clutch reminder opens its detail screen, via `navigationRef` (`src/navigation/navigationRef.ts`) and `handleNotificationResponse()` (`src/services/notificationNavigation.ts`), which handles both a warm tap (`addNotificationResponseReceivedListener`) and a cold start from a killed app (`getLastNotificationResponseAsync`). See §5.17.

5. **Preserve existing architecture** (repository pattern, one-hook-per-data-shape, versioned single-file migrations, feature-hub navigation) **unless a change is specifically required and explained.**
6. **Make small, controlled changes** scoped to the requested phase/sprint — per constitution §19, "do not add major unrequested features."
7. **Test changes before marking work complete.** At minimum: `tsc --noEmit`, `expo-doctor`, `expo export`, and the unused-import sweep. For UI changes, actually run the app and exercise the feature — type-checking is not feature verification.
8. **Update this file (`PROJECT_CONTEXT.md`) after major features or architectural changes** — keep §5 (Implemented), §6 (Partial), §7 (Pending), and §10 (Known Issues) current.
9. **Never store secrets in this file.** If environment variables are introduced in the future, document them by name only in §11.
10. **When something in this document conflicts with the actual code, trust the code** — this file is a snapshot, verified as of the date at the top; it decays over time.

## 14. Backend — Secure AI Gateway (Phase 3 Sprint 3.1)

### 14.1 Why this exists

`SECURITY.md` Rules 3–5 require that no paid AI provider (Gemini/OpenAI/Claude/etc.) is ever called directly from the mobile client with a secret key, and that all such requests are proxied through a backend holding the secret server-side. This backend is that gateway. It was built **before** any real provider integration specifically so the request/response architecture, validation, and abuse-protection scaffolding could be established and reviewed while the security stakes were still zero (no real key, no billing risk, no real user data in transit).

### 14.2 Architecture

```
Flockora App (Expo/RN)  ── HTTPS (not yet wired up) ──▶  Flockora Backend API  ──▶  AI Provider (not yet integrated)
```

- **Location:** [`flockora-backend/`](flockora-backend/) — a sibling directory to `flockora-app/` in the same repository, not a separate repo.
- **Stack:** Node.js + TypeScript (`strict: true`) + Express 4, `zod` for schema validation, `multer` (v2.x) for multipart file handling, `helmet` for security headers, `express-rate-limit` for rate limiting, `dotenv` for local env loading.
- **Entry point:** [`src/index.ts`](flockora-backend/src/index.ts) starts the server; [`src/app.ts`](flockora-backend/src/app.ts) assembles the Express app and defines middleware order (security headers → CORS → request ID → timeout guard → body size limits → routes → 404 handler → centralized error handler, in that order — the error handler must be last per Express convention).
- **Provider abstraction** ([`src/providers/`](flockora-backend/src/providers/)): `AIProvider` is the interface every provider implements (`analyzeBird(input): Promise<AnalyzeBirdResult>`). `MockAIProvider` is the only implementation that exists today — deterministic, offline, per-species canned responses, no network call, no file bytes retained. `providerFactory.ts` selects the active provider from the `AI_PROVIDER` env var (only `"mock"` is a valid value right now — the zod schema in `src/config/env.ts` will reject anything else at startup). **Adding a real provider (e.g. `GeminiProvider`) in a future sprint means adding one new file plus one new `case` in `providerFactory.ts` and one new enum value in `env.ts` — nothing in `src/routes/` or the mobile client's contract needs to change.**
- **Response shape** deliberately mirrors the client's existing `AIAnalysisResult` type (`flockora-app/src/types/onboarding.ts`: `breed`/`sex`/`color`/`lifeStage`, each `{ value, confidence: 'HIGH'|'LIKELY'|'UNSURE' }`) so that wiring the client to this backend later is a data-source swap, not a UI/type rework.

### 14.3 Endpoint structure

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/health` | Liveness check, no auth, returns `{ status: 'ok' }` |
| `POST` | `/api/v1/analyze-bird` | The only AI-facing endpoint. `multipart/form-data`: `photo` (file, required, JPEG/PNG/WEBP only, size-capped via `MAX_UPLOAD_MB`), `speciesHint` (string, optional, validated against the 8-species enum). Returns `{ requestId, provider, result }` on success. |

Every response — success or error — includes an `X-Request-Id` header and a matching `requestId` field in the JSON body, generated per-request server-side (`src/middleware/requestId.ts`), for traceability without needing to log anything sensitive.

### 14.4 Security boundary

- The mobile client **cannot** call an AI provider directly today — it has no AI provider code of any kind, mocked or real, and makes zero network calls (unchanged from before this sprint).
- The backend **cannot** call a real AI provider today either — only `MockAIProvider` exists; no provider SDK, no API key, no network call to any external AI service exists anywhere in `flockora-backend/`.
- No secret of any kind exists in this codebase. `flockora-backend/.env.example` documents variable *names* only (`PORT`, `NODE_ENV`, `AI_PROVIDER`, `MAX_UPLOAD_MB`, rate-limit/timeout/CORS settings) — no values that could be mistaken for real secrets, and it explicitly documents that future provider keys must never be added to this file with real values (per `SECURITY.md` Rules 1, 2, 5).
- `flockora-backend/.gitignore` excludes `.env`, `.env.*` (with an explicit `!.env.example` exception), `node_modules/`, and `dist/` — verified via `git add --dry-run` that none of these are staged.
- **Rate limiting is IP-based only** (`express-rate-limit`, applied to `/api/v1/analyze-bird`), because no authentication exists anywhere in Flockora yet — there is no user identity to key a per-user quota on. This is explicitly documented as incomplete against `SECURITY.md` Rule 7 (see §14.5) — it is not a substitute for per-user/global usage controls, which are required before a real, billable provider is connected.
- Logging (`src/utils/logger.ts`) is type-constrained to a fixed, narrow field set (`requestId`, `route`, `statusCode`, `provider`, `mimeType`, `sizeBytes`, `code`) — there is no code path by which a request body, uploaded file, header value, or future auth token could be logged. Verified by manual review of every `logger.*` call site and by running the server locally and inspecting actual log output.

### 14.5 Mocked provider status — what's real vs. simulated

| Aspect | Status |
|---|---|
| HTTP server, routing, middleware chain | **Real** — genuinely running, tested locally (start, `/health`, valid upload, invalid MIME, invalid species hint, oversized file, 404, missing file) |
| Request validation (MIME allowlist, size limit, species enum) | **Real** — enforced by `multer` + `zod`, verified to reject bad input with correct 400/413-equivalent responses |
| Rate limiting | **Real mechanism, incomplete policy** — actual `express-rate-limit` middleware is active and returns `RateLimit-*` headers and 429s, but is IP-only (see §14.4) |
| Request timeout | **Real mechanism, untested under real latency** — the middleware is wired and would fire, but the mock provider resolves near-instantly, so this path has not been exercised end-to-end against genuine slow-provider latency |
| Centralized error handling | **Real** — verified for validation errors, multer errors, and the 404 path |
| AI analysis result | **Fully simulated** — `MockAIProvider` returns a hardcoded lookup table keyed by `speciesHint`, defaulting to `chicken`. No image is actually analyzed; the uploaded file's *bytes* are validated (type/size) but never inspected for content. |
| Environment variable validation | **Real** — `src/config/env.ts` uses a `zod` schema and calls `process.exit(1)` with a safe (names-only) error message if required vars are missing/invalid; verified by the app failing to start against a malformed `.env` during manual testing |

### 14.6 Remaining work before real AI integration

Explicitly not done in this sprint, required before `MockAIProvider` is replaced with a real provider:

1. **A real `AIProvider` implementation** (e.g. `GeminiProvider`) calling an actual AI API, with its secret key injected only via the deployment platform's environment/secrets store (`SECURITY.md` Rule 5) — never committed, never in `.env.example` with a real value.
2. **Per-user usage quotas** (`SECURITY.md` Rule 7) — requires an authentication system to exist first (nothing in Flockora has user identity today); the current IP-based rate limiter is not a substitute.
3. **A global usage ceiling / circuit breaker** (`SECURITY.md` Rule 7) to cap total spend/volume across all users.
4. **Deployment** — no hosting platform has been chosen or configured for this backend yet; it currently only runs locally.
5. ~~**Client integration**~~ — **done in Sprint 3.2** (see §14.7). The mobile app's onboarding flow now calls this backend for real; item retained here only for history.
6. **A fresh, dedicated security audit** (`SECURITY.md` Rule 15) — mandatory immediately after real AI integration, and separately if authentication is added first.
7. **Deeper abuse protections** appropriate to a live, billable endpoint — e.g. request signing/attestation from the mobile client, image-content sanity checks, and CAPTCHA-equivalent protections if abuse patterns emerge — none of which are meaningful to design until a real provider and real traffic exist.

### 14.7 Mobile client integration (Phase 3 Sprint 3.2)

**What changed, by file:**
- [`src/services/imagePickerService.ts`](flockora-app/src/services/imagePickerService.ts) (new) — the only place `expo-image-picker` is called. `captureFromCamera()` and `pickFromGallery()` each request their specific permission lazily (only when invoked, never on screen mount) and return a discriminated `PickPhotoOutcome`: `success` (with a `{ uri, mimeType, fileName }` photo), `canceled`, or `permission_denied` (with `canAskAgain` so the UI can offer "Open Settings" only when the OS says re-prompting won't work).
- [`src/services/birdAnalysisService.ts`](flockora-app/src/services/birdAnalysisService.ts) (new) — the **only file in the entire mobile client that calls `fetch`**. `analyzeBirdPhoto()` builds a `multipart/form-data` request (the `photo` file plus an optional `speciesHint` derived from the onboarding species selection), posts it to `${getBackendBaseUrl()}/api/v1/analyze-bird` with a 20-second client-side `AbortController` timeout, and returns a discriminated `BirdAnalysisOutcome`: `success` (with the analysis result, typed identically to the client's existing `AIAnalysisResult`) or `error` (with a `kind` — `invalid_image | payload_too_large | rate_limited | timeout | network | server_error | unknown` — derived from the backend's own safe `code`/`message` response fields, or from a local `fetch` failure/`AbortError` when there's no response at all).
